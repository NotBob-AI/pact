# PACT Security Notes

## Attack Surface Distinctions

PACT receipts protect **agent-initiated** tool calls. The interceptor sits between the agent and the MCP server — it sees every tool call the agent makes and generates a receipt proving the call was within committed policy.

PACT does NOT protect against:

### 1. Auth-bypass attacks on the MCP server (BadHost class)

**Attack:** Attacker bypasses authentication at the transport layer (e.g., CVE-2026-48710 BadHost — Starlette HTTP Host header auth bypass). The MCP server processes requests without a valid agent session.

**PACT posture:** The interceptor never sees these calls — the agent isn't involved, so no receipt is generated. This is a separate security domain from PACT's scope.

**Implication:** PACT receipts are necessary but not sufficient for AI agent security. They prove what the agent did; they do not prove what an attacker did after bypassing the agent.

### 2. Credential store exfiltration via MCP server compromise

**Attack:** MCP server is compromised through another vector (supply chain, config error). Attacker uses stored credentials directly, bypassing the agent.

**PACT posture:** No agent action = no receipt. The credential store is outside PACT's trust boundary.

**Implication:** MCP credential stores need their own hardening (scoped tokens, time-limited grants, hardware key storage). PACT receipts on tool calls don't prevent credential theft.

### 3. The compounding effect

When BadHost + credential store exfiltration combine: the attacker bypasses auth AND uses the MCP server's own stored credentials. From the MCP server's perspective, the requests are authenticated (they came from a valid session via the compromised credential). PACT receipts would show a clean log — because the receipts are generated by the interceptor, which wasn't in the path.

## What PACT does prove

PACT generates receipts for agent-initiated actions. The receipts are meaningful precisely because they are generated below the agent layer — the agent cannot falsify them.

What a PACT receipt proves:
- `tool_called ∈ committed_policy` at the moment of execution
- The action was initiated by the agent (not an external attacker)
- The action happened within the agent's scoped permission boundary
- The action is logged to a transparency log that the agent cannot rewrite retroactively

What a PACT receipt does NOT prove:
- That the MCP server's credential store was not compromised
- That no auth-bypass attack occurred on the same infrastructure
- That the agent's execution environment was not tampered with by a separate attacker

## Architectural Implication

**Layer 0 receipts ≠ MCP server hardening.**

The distinction matters for deployment: PACT is an accountability layer for agent actions, not a transport security layer for MCP infrastructure. Both are needed; neither is sufficient alone.

This aligns with OWASP MCP Top 10 categories:
- MCP08 (Lack of Audit/Telemetry) → PACT addresses this
- MCP05 (Command Injection), MCP02 (Rug Pull), MCP03 (Tool Poisoning) → separate attack surfaces, not covered by PACT receipts alone
